# The Slow Test Suite

**Problem:** `sw-01`  
**Model:** `claude-haiku-4-5`  
**Prompt tier:** none  
**Score:** 3/5

## Scenario

A team's CI pipeline takes 45 minutes. Developers start batching commits to avoid waiting, leading to larger PRs and more merge conflicts. The team is asked to speed up the test suite. They parallelize it, decomposing integration tests into isolated unit tests, and get it down to 12 minutes. Three months later, the defect rate has doubled.

**Question:** Why did defects increase after making CI faster?

## Model Response

# Why Defects Doubled After Faster CI

The most likely culprit: **They replaced integration tests with unit tests, losing critical coverage.**

## The Core Problem

When they "decomposed integration tests into isolated unit tests," they eliminated tests that catch bugs unit tests can't find:

- **Component interactions** – How services communicate with each other
- **Configuration errors** – Mismatched settings between environments  
- **Database/persistence issues** – Schema changes, transaction boundaries
- **Timing & race conditions** – Real concurrency problems
- **API contracts** – Version mismatches, contract violations
- **Environmental surprises** – Things that only fail in production

Unit tests validate logic in isolation. Integration tests validate the *system works together*.

## Secondary Issues

1. **False confidence**: Faster feedback loops psychologically lower developers' guard. "It passes CI in 12 minutes, so it must be fine."

2. **Changed behavior**: With faster CI, developers now commit more frequently (good!), but they may skip local integration testing or spend less time thinking through edge cases.

3. **Regression in rigor**: Code reviews might become less thorough if reviewers believe the automated checks are comprehensive.

## The Real Lesson

This is a classic optimization trap: **they optimized for the wrong metric (speed) at the cost of the right one (quality).**

A 45-minute suite catching 95% of bugs is better than a 12-minute suite catching 70%.

## What Should Have Happened

- Parallelize the *full* test suite across machines (not eliminate integration tests)
- Use tiered testing: fast unit tests first, then full integration suite in parallel
- Understand *what kind* of defects are appearing—that tells you what test coverage was lost
